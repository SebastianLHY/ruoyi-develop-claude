## 🎯 版本号规范

### 语义化版本格式

```
MAJOR.MINOR.PATCH

示例: v1.0.1
```

### 版本号含义

| 位置 | 名称 | 何时递增 | 示例 |
|------|------|---------|------|
| **MAJOR** | 主版本号 | 不兼容的API修改、重大架构调整 | v1.0.0 → v2.0.0 |
| **MINOR** | 次版本号 | 向下兼容的功能性新增 | v2.17.0 → v2.18.0 |
| **PATCH** | 修订号 | 向下兼容的问题修正、小优化 | v2.18.0 → v2.18.1 |

## 🔄 版本发布流程

### 1. 确定版本类型

```bash
# MAJOR版本（重大变更）
- 不兼容的API修改
- 架构重构
- 核心流程重大调整

# MINOR版本（功能新增）
- 新增智能体
- 新增核心功能
- 优化现有流程

# PATCH版本（问题修复）
- Bug修复
- 文档更新
- 小幅优化
```

### 2. 更新版本号

需要更新以下文件：

```
主文档：
- dev.md (第3行)
- QUICK-START.md (第3行)
- VERSION.md (第3行)

详细文档：
- dev-steps.md (第3行)
- dev-rollback-guide.md (第3行)
- dev-dependency-check.md (第3行)
- 其他相关文档
```

### 3. 更新版本历史

在以下文件中更新版本历史：
- `dev.md` - 版本历史表格
- `VERSION.md` - 完整版本历史

### 4. 归档旧版本（可选）

如需归档重要版本：
```bash
# 创建归档目录
mkdir -p archives/v1.0.0

# 归档重要文档
cp dev.md archives/v1.0.0/
cp CHANGELOG.md archives/v1.0.0/
```

## 📝 变更日志规范

### 变更类型

| 类型 | 说明 | 标记 |
|------|------|------|
| **Added** | 新增功能 | ✅ |
| **Changed** | 功能变更 | 🔄 |
| **Deprecated** | 即将废弃 | ⚠️ |
| **Removed** | 已删除 | ❌ |
| **Fixed** | Bug修复 | 🐛 |
| **Security** | 安全修复 | 🔒 |

### 变更日志模板

```markdown
## [v1.0.1] - 2026-02-01

### Added ✅
- 新增功能描述

### Changed 🔄
- 功能变更描述

### Fixed 🐛
- Bug修复描述

### Performance ⚡
- 性能优化描述
```

## 💡 版本管理最佳实践

1. **版本号统一**: 所有文档使用相同的版本号
2. **变更记录**: 每次发版都要更新CHANGELOG
3. **向下兼容**: MINOR和PATCH版本必须向下兼容
4. **文档同步**: 版本发布时同步更新所有相关文档
5. **归档管理**: 重要版本及时归档到archives/目录
